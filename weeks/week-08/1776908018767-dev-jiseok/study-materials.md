## 전체 분석 프레임워크 (최종 정리)

### 1단계: 문제 파악
| 파일 | 분석 내용 |
|------|----------|
| README.md | 회사 배경, 인프라 규모, 성장 단계 |
| main.tf | 리소스 설정에서 낭비/과잉 포인트 식별 |

### 2단계: 증거 수집
| 파일 | 분석 내용 |
|------|----------|
| metrics.json | 30일 시계열 패턴 — 피크/평균/트렌드 |
| cost_report.json | 6개월 서비스별 비용 추이 |
| business_metrics.json | 일별 트래픽/주문수/단위경제 |
| tags_inventory.json | 태그 커버리지, 비용 할당 사각지대 |
| cloudwatch_format.json | CloudWatch export 형식 메트릭 |
| cur_report.csv | 라인 아이템 레벨 비용 분석 |
| ri_sp_coverage.json | RI/SP 커버리지, 예약 활용률 |

### 3단계: 솔루션 작성
1. **문제 식별**: 어떤 리소스가 왜 문제인지
2. **근본 원인**: 설계 오류/설정 미비/프로세스 부재
3. **해결책**: solution.tf + 운영 프로세스 개선
4. **절감액 추정**: 월간/연간 절감액 + 계산 근거
5. **단위경제 분석**: cost_per_order, cost_per_1k_requests 변화
6. **알림 설계**: 재발 방지를 위한 CloudWatch 알림

## 자주 나오는 실수와 개선 포인트

### 감점 패턴 Top 5
1. **시나리오 불일치**: main.tf를 안 읽고 다른 문제 해결책 제출
2. **decoy 리소스 삭제**: 정상 리소스까지 제거 (SourceVolumeStatus=exists 등)
3. **절감액 과대/과소 추정**: 정답 ±30% 이내가 목표
4. **리포트 미제출**: solution.tf만 제출하면 근본 원인/절감액 점수 0점
5. **리소스 삭제 vs 설정 수정**: ECR 삭제가 아닌 lifecycle 추가가 정답

### 고득점 패턴
1. **main.tf + metrics.json + cost_report.json 교차 분석**
2. **정확한 절감액 계산** (AWS 공식 단가 기반)
3. **decoy 리소스 올바르게 유지**
4. **단위경제 분석 포함** (L2+)
5. **예방 조치 포함** (DLM, lifecycle, 알림 설정)

## 최종 발표 가이드

### 발표 구성 (10분)
1. **분석 요약** (2분): 발견한 문제 목록, 총 절감액
2. **핵심 사례 Deep Dive** (5분): 가장 임팩트 큰 문제 1-2개 상세 분석
3. **단위경제 개선** (2분): 최적화 전후 cost_per_order 변화
4. **거버넌스 제안** (1분): 재발 방지 프로세스

### 발표 팁
- 숫자로 말하기 (월 $X 절감, 연 $Y 절감)
- Before/After 비교표 활용
- "왜 이게 문제인지"를 비즈니스 관점에서 설명
- 실행 가능한 다음 단계(Next Steps) 포함
